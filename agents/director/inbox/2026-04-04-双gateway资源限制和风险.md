## 文档概要

### 一、飞书"不回弹 / 仍可卡片"问题
| 方向 | 要点 |
|------|------|
| 群聊触发 @ | `groups["*"].requireMention: true` 导致未 @ 机器人直接回复2次 |
| 日常位置 | 各自 LaunchAgent 写不同日志路径 |
| 长连接不稳定 | 大量 `[ws] reconnect` / `timeout 15000ms` 导致假死 |
| 匹配 / DM | `dmPolicy: open` 被群聊干扰，18789 与 18889 冲突 |

### 二、多 Bot 同时连 OpenClaw"撞上"——机制与修改可能
- **多路 WebSocket**：共 9 Node - 7-14 循环，负载叠加
- **队列按 chatId**：不按 token 区分
- **冗余 probe**：多账号预检 + 长连接 WS；上游有讨论 probe 超限与 mention 频率 (PR/issue)
- **已落地的配置**：`messages.inbound` debounce 1500ms / 2000ms；`maxConcurrent 24` / `subagents.maxConcurrent 12`
- **未落地的受限版本**：queue / maxConcurrent / steer-collect；`agents.list[].maxConcurrent` 全局 openclaw 未记；reconnectDelayMs

### 三、ClawHub / Skills
- `clawhub` CLI：registry 限流，部分 skill 有 VirusTotal 提示
- **解决方案**：将 skill 与多 Bot 隔离，提升 Agent 写法与运营文化

### 四、双 Gateway（单机 + Docker 分布式）——方案与风险
**目的设想：**
- Gateway-1（单机 18889）：`da_jie_tou_win` 等 → director / scout / auditor
- Gateway-2（Docker 新建）：`mei_gu_zuo_kong_win` 等 → architect / censor / writer
- 隔离：Docker 独享进程序与环境

**风险：**
- 内存/CPU 限制上尤
- 冲突"实验/生产"与机器分离
- 跨 Gateway 的 agentToAgent / 统一一话需额外配置
